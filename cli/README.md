# ArduPilot SITL Parameter Identification CLI

Command-line interface for identifying Flight Dynamics Model parameters from ArduPilot DataFlash logs.

## Installation

From the project root:

```bash
pip install -r requirements.txt
```

## Usage

### Basic Usage

```bash
python -m cli.sysid \
    --log flight.bin \
    --frame quad_x \
    --mass 1.2 \
    --output sitl_params.parm
```

### Full Options

```bash
python -m cli.sysid \
    --log flight_2024_01_15.bin \
    --frame quad_x \
    --mass 1.2 \
    --mass-std 0.05 \
    --arm-length 0.165 \
    --arm-length-std 0.005 \
    --prop-diameter 5.1 \
    --motor-kv 2300 \
    --output sitl_params.parm \
    --report results.json \
    --verbose
```

### Excitation Check Only

Check if your flight data has sufficient excitation for parameter identification:

```bash
python -m cli.sysid \
    --log flight.bin \
    --frame quad_x \
    --mass 1.2 \
    --output dummy.parm \
    --excitation-check-only
```

## Options

### Required Options

- `--log PATH`: Path to ArduPilot .bin log file
- `--frame TYPE`: Vehicle frame type (quad_x, quad_plus, hexa_x, octo_x)
- `--mass FLOAT`: Vehicle mass in kg (measured)
- `--output PATH`: Output .parm file path

### Optional Physical Priors

These help regularize the optimization:

- `--mass-std FLOAT`: Mass measurement uncertainty in kg (default: 0.05)
- `--arm-length FLOAT`: Motor arm length in meters (default: 0.165)
- `--arm-length-std FLOAT`: Arm length uncertainty in meters (default: 0.005)
- `--prop-diameter FLOAT`: Propeller diameter in inches (default: 5.1)
- `--motor-kv FLOAT`: Motor KV rating (default: 2300)

### Optional Output

- `--report PATH`: Generate JSON report with detailed results
- `--verbose`: Show detailed progress and diagnostics
- `--excitation-check-only`: Only check excitation, don't optimize

### Other

- `--version`: Show version number
- `--help`: Show help message

## Pipeline Stages

The CLI orchestrates the full identification pipeline:

1. **Parse DataFlash Log**: Extract IMU, GPS, RCOUT, EKF data
2. **Preprocessing**: Timestamp alignment, resampling to 400 Hz
3. **EKF Health Segmentation**: Filter segments with good EKF health
4. **State Estimation**: RTS smoother for optimal state trajectory
5. **Excitation Analysis**: Check if data can identify all parameters
6. **MAP Optimization**: Identify parameters with physical priors
7. **Hold-out Validation**: Test on unseen flight segment
8. **Output Generation**: Write .parm and optional JSON report

## Troubleshooting

### "No healthy EKF segments found"

Your log needs good GPS lock and EKF health. Try:
- Flying in an area with better GPS reception
- Ensuring the vehicle had time to achieve GPS lock before takeoff
- Checking that EKF health was good during flight (visible in logs)

### "Data quality is POOR"

The flight data lacks sufficient excitation for some parameters. Use:
```bash
python -m cli.sysid --log flight.bin --frame quad_x --mass 1.2 --output dummy.parm --excitation-check-only
```

This will tell you which maneuvers to fly to excite poorly-identified parameters.

### Import Errors

Make sure you're running from the project root:
```bash
cd /path/to/differentiable-sitl
python -m cli.sysid --help
```

## Requirements

- Python 3.8+
- See `requirements.txt` for package dependencies
- Valid ArduPilot DataFlash .bin log file
- Log must contain: IMU, RCOUT (motor outputs), and ideally GPS + EKF data
